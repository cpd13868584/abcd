# design-package contract (package-spec)

Every abcd command produces / consumes this one standard structure. **Humans** read `rendered/*.svg`; **AI** reads the `.mmd` / `.puml` source + `manifest.json`. One source, two consumers.

## Directory structure

```
design/
  README.md                 # index: one-line goal + a tour of each diagram (entry point for an AI consumer)
  index.html                # self-contained viewer (auto-generated by /abcd-model / /abcd-map; re-render via /abcd-view)
  manifest.json             # machine-readable map (schema below) — single source of truth / metamodel layer
  00-glossary.md            # ubiquitous language / core-domain glossary
  A-business/               # forward: business modeling
    vision.md               # vision: boss + target organization + quantified improvement goal
    usecases.puml           # business use-case diagram (PlantUML)
    sequence-as-is.mmd      # business sequence diagram · as-is
    sequence-to-be.mmd      # business sequence diagram · to-be
  B-requirements/           # forward: requirements
    usecases.puml           # system use-case diagram (PlantUML)
    specs/<usecase>.md            # use-case spec (structured text, single source, all fields)
    specs/<usecase>.activity.puml # activity diagram mechanically generated from the path steps (view)
  C-analysis/               # forward: analysis
    domain.mmd              # analysis class diagram / domain model
    states/<entity>.mmd     # entity state machine (optional)
  recovered/                # reverse: design-level as-built (physically separate from forward)
    class.mmd                        # data model (entities + fields + FKs, from schema)
    <scope>.oo.mmd                   # distilled OO class diagram (attributes + operations, optional)
    architecture.mmd
    sequences/<flow>.mmd             # system sequence diagram (system ↔ external systems/services)
    sequences/<flow>.design.mmd      # design sequence diagram (object/method level, optional)
    business-sequence-skeleton.mmd   # business-sequence skeleton (gap-marked, for /abcd-model to complete)
  rendered/*.svg            # rendered diagrams (for humans; AI reads the source, no rendered needed)
```

**Why `A-/B-/C-` and `recovered/` are physically separate**: it puts the discipline "requirements ≠ design, forward ≠ reverse" into the directory layout. Reverse artifacts live only in `recovered/` and never masquerade as the forward requirements/business model.

## `manifest.json` schema

```jsonc
{
  "system": "system name",
  "purpose": "one line: why this system exists",
  "generated": "ISO time",

  "vision": {                              // A layer, forward-only source
    "boss": "boss (concrete name + title)",
    "organization": "target organization",
    "goals": [{ "metric": "measurable org-level improvement metric", "baseline": "", "target": "" }]
  },
  "glossary": [{ "term": "", "definition": "", "core_domain": true }],

  "diagrams": [
    {
      "id": "biz-seq-checkout-tobe",
      "workflow": "A",                     // A|B|C|D
      "type": "business-sequence",         // usecase|business-usecase|business-sequence|system-sequence|design-sequence|class|data-model|state|er|architecture
      "tool": "mermaid",                   // mermaid|plantuml
      "provenance": "forward",             // forward|reverse|hybrid
      "level": "business",                 // business|requirement|analysis|design
      "phase": "to-be",                    // as-is|to-be (business sequence only)
      "confidence": 1.0,
      "gaps": [],                          // reverse-skeleton / hybrid gaps, e.g. ["manual worker unknown","as-is incomplete"]
      "file": "A-business/sequence-to-be.mmd",
      "rendered": "rendered/biz-seq-checkout-tobe.svg",
      "title": "Checkout (to-be)",
      "covers": ["uc-checkout"],           // related use-case ids
      "code_refs": [
        { "symbol": "CheckoutController.submit", "path": "src/checkout/controller.ts", "lines": "12-88" }
      ]
    }
  ],

  "use_cases": [
    { "id": "uc-checkout", "actor": "Customer", "name": "Checkout",
      "spec": "B-requirements/specs/checkout.md",
      "realized_by": ["Order", "Payment"],     // analysis classes that realize it
      "code_refs": [ /* same shape as above */ ] }
  ],

  // three-link traceability: use case → analysis class → code symbol (business↔code navigation both ways)
  "traceability": [
    { "use_case": "uc-checkout", "analysis_classes": ["Order", "Payment"], "code": ["src/order/", "src/payment/"] }
  ],

  // honest coverage: flows that exist in the code but this package did NOT recover.
  // Keeps a partial reverse map from reading as the whole; rendered in the HTML viewer.
  "uncovered_flows": [
    { "node": "Refund", "reason": "not traced this pass", "code_hint": "src/refund (refund-create / refund-settle)" }
  ],

  // forward artifacts can feed an implementation task directly; sync verifies AI output against acceptance
  "ai_spec": { "is_implementation_input": true, "constraints": [], "acceptance": [] }
}
```

## Three hard fields (get them right)

- **`provenance`** = `forward` | `reverse` | `hybrid`
- **`level`** = `business` | `requirement` | `analysis` | `design`
- **`gaps`** = the gap list of a reverse-skeleton / hybrid artifact (manual steps, missing as-is, parts pending dialogue, etc.)

Placement rules:
- Pure reverse artifact: `provenance=reverse`, `level=design`, with `gaps`, in `recovered/`.
- Forward artifact: `provenance=forward`, in `A-/B-/C-`.
- Business sequence diagram completed by dialogue: `provenance=hybrid`, in `A-business/`.
- **The requirements layer (vision / system use case / use-case spec) is never `reverse`.**
- **Reverse declares what it did not cover.** List flows that exist in the code but weren't recovered in `uncovered_flows`, so a partial map never reads as the whole.

## Three-link traceability

`use case → analysis class → code symbol`, carried by `manifest.traceability` plus each `diagram`/`use_case`'s `code_refs`. This lets a consumer (human or AI) drill from a business use case down to code, and trace from code up to the business value it serves. This is the technical realization of "business / tech / code legible at a glance".
